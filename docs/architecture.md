# 架构设计（Architecture）

> xu-bazaar PVE 卡牌对战游戏的技术栈与分层架构设计。

---

## 1. 技术栈

| 层 | 选型 | 说明 |
| --- | --- | --- |
| 构建 | **Vite** | 快速开发服务器与构建。 |
| UI 框架 | **React + TypeScript** | 组件化 UI，类型安全。 |
| 动画 | **Framer Motion（motion）** | 负责插入/重排/进出场动画。 |
| 状态管理 | **Zustand** | 轻量状态管理，桥接战斗引擎与 UI。 |
| 单元测试 | **Vitest** | 覆盖纯 TS 战斗引擎的关键规则。 |

### 1.1 为何选择 React + Framer Motion

React 本身不负责动画，只管理「状态 → DOM 结构」。动画交由 Framer Motion 处理：

- **插入式召唤 / 重排让位**：使用 `layout` 属性 + `AnimatePresence`，元素增删/重排会自动补间。仆从进场、死亡移除、格子左移填补全部自动过渡，几乎零手写动画。
- **性能**：卡牌游戏为中低频、元素数量少（数十个 DOM 节点）的场景，远达不到 React 性能瓶颈。DOM + Framer Motion 完全够用，开发效率与可维护性高于 Canvas。
- 若未来需要大量粒子/精灵特效，可考虑 PixiJS 渲染战场 + React 做 UI 面板的混合方案，但第一版不做此过度设计。

## 2. 分层架构

```
src/
  engine/   # 纯TS战斗引擎（状态机、结算、AI）—— 不依赖 React/UI
  data/     # 卡牌/仆从定义数据
  store/    # zustand，桥接 engine 与 UI
  ui/       # React组件（战场、手牌、仆从、动画、指向箭头等）
```

### 2.1 engine（战斗引擎）
- **纯 TypeScript，不依赖 React 或任何 UI 库。**
- 负责：回合状态机、抽牌/疲劳、能量、出牌结算、自动战斗结算、胜负判定、敌人 AI。
- 以纯函数 / 不可变状态转换为主，便于单测与回放。
- 对外暴露：初始状态构造、事件/动作处理（如「出牌」「结束回合」「执行自动战斗」）、返回新状态与一系列可供 UI 播放动画的「战斗事件（BattleEvent）」序列。

### 2.2 data（数据定义）
- 卡牌、仆从原型、示例卡组等静态数据。
- 与 engine 解耦，engine 通过数据 ID 引用。

### 2.3 store（状态桥接）
- 使用 Zustand 持有当前战斗状态。
- 接收 UI 动作，调用 engine 计算新状态与事件序列。
- 驱动 UI 逐步播放 engine 产出的战斗事件（动画）。

### 2.4 ui（渲染层）
- React 组件：战场、敌我角色区、仆从区、手牌、中线（能量/回合箭头/结束回合按钮）、指向箭头、左侧战斗日志（可展开/收起）、占位元素（装备槽/遗物/场地效果）；英雄技能按钮（有技能时可交互）。
- 只读取 store 状态并派发动作，不包含战斗规则逻辑。

---

## 3. 核心原则

- **战斗核心逻辑必须为纯 TS、不依赖 UI**，以便单测与未来复用（PVE 关卡、成长系统）。
- **规则集中在 engine**，UI 不得实现战斗规则。
- **状态驱动**：UI 是状态的函数，动画由 engine 产出的事件序列驱动播放。

## 4. 未来扩展点（预留，不在第一版实现）

| 方向 | 预留设计 |
| --- | --- |
| 战斗外成长 | engine 与角色/卡组数据分离，成长系统可在战斗外修改角色基础属性、卡组内容，再注入战斗初始状态。 |
| 卡牌解锁 | data 层维护全量卡池与「已解锁」集合，构筑系统从已解锁集合中选卡。 |
| 装备系统 | 装备/武器非卡牌；角色 `equipmentSlot` 与 UI 占位；engine 在结算钩子处预留效果介入点。 |
| 遗物 / 场地效果 | 预留角色遗物列表、场地效果字段与结算钩子；第一版仅占位。 |
| 英雄技能 | `HeroDef.skill` + `useSkill`；词条 `onKill` / `draw`；默认玩家地狱术士、敌人训练假人。 |
| 关键字扩展 | 状态词条（嘲讽）与条件/效果词条可分别扩展。 |
