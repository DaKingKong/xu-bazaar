# 战斗日志设计（Battle Log）

> 本文档定义战斗日志的**展示内容、文案规则与 UI 行为**。
> 实现须复用现有 `BattleEvent` 序列（见 `data-model.md` §7 / `architecture.md` §2.3），不在 UI 层重新推算规则。
> **实现状态（已落地）**：`src/store/formatLog.ts` + store 播放追加；UI 为左侧可折叠面板（`BattleLogPanel`）。

---

## 1. 目标

战斗日志让玩家在动画播放之余，能**回看本局发生过什么**：谁出了什么牌、谁打了谁、谁死了、阶段如何切换。

| 目标 | 说明 |
| --- | --- |
| 可追溯 | 关键结算都能在日志中找到对应条目 |
| 可读 | 用人名/卡名/角色称呼，避免「某个仆从」「造成 N 点伤害」这类含糊句 |
| 不干扰操作 | 日志是辅助信息，不挡手牌与战场交互 |
| 与动画同源 | 条目在 store **播放该 `BattleEvent` 时**追加，时间线与画面一致 |

**非目标（本版不做）：**

- 可点击跳转到历史状态 / 时间轴回放
- 导出战报、分享截图模板
- 把日志当唯一反馈（飘字、血条、阶段箭头仍保留）

## 2. 数据流

```
engine 结算 → BattleEvent[]
       ↓
store 逐条播放（applyEventToView + 动画延迟）
       ↓
同一时刻：formatLog（可向前窥视队列）→ 追加 LogEntry（或静默）
       ↓
UI 只渲染 log 列表（只读）
```

**配对规则：** 自动战斗里一次交锋在引擎中是 `attack` +（可选）紧随的 `counter`。日志把它们看成**一次战斗事件**，只出**一条**文案；`counter` 本身不再单独成条。动画仍可按两条 `BattleEvent` 依次播放（突进 / 飘字），仅日志合并。

### 2.1 LogEntry（建议形状）

```ts
interface LogEntry {
  id: number;                 // 会话内递增，作 React key
  text: string;               // 最终展示文案（中文）
  kind: LogKind;              // 用于样式/过滤（见 §4）
  side?: Side;                // 主动方（便于着色）；阶段/胜负可省略
  sourceEvent: BattleEvent['type']; // 合并条用 'attack'
}

type LogKind =
  | 'phase'      // 阶段切换
  | 'resource'   // 抽牌 / 跳过 / 能量重置
  | 'play'       // 出牌 / 召唤
  | 'combat'     // 交锋（attack[+counter]）/ 治疗 / 死亡
  | 'fatigue'    // 疲劳
  | 'system';    // 胜负等
```

格式化函数放在 **store 侧**（或 `src/store/formatLog.ts`），需要查 `cardDb` / 场上仆从名时读**播放前或播放后的 view**（见 §3.3 命名解析）。engine 不产出文案字符串，保持纯规则层。

### 2.2 静默事件

下列事件**默认不写日志**（画面已有足够反馈，写满会刷屏）：

| 事件 | 原因 |
| --- | --- |
| `energyReset` | 每回合固定重置 4，中线能量条已表达 |
| `draw` / `drawSkipped` | 手牌区已可见；抽牌信息不写入日志 |

**本版约定：** `energyReset`、`draw`、`drawSkipped` 静默。

## 3. 文案规则（按 BattleEvent）

### 3.1 称呼约定

| 对象 | 文案 |
| --- | --- |
| `side === 'player'` | **你** |
| `side === 'enemy'` | **敌人** |
| 己方英雄 | **你**（或「你的英雄」，攻击/治疗指向英雄时用「你」） |
| 敌方英雄 | **敌人** |
| 仆从 | **「{minion.name}」**（来自 `Minion` / `CardDef.minion.name`） |
| 卡牌 | **「{CardDef.name}」** |
| 疲劳生成卡 | **「疲劳突袭」**（或当前 `FATIGUE_STRIKE` 的正式名） |

攻击/目标写成：`你的「突击兵」` / `敌人的「铁壁守卫」` / `你` / `敌人`。

### 3.2 事件 → 文案模板

| `BattleEvent` | 是否记录 | 模板（示意） |
| --- | --- | --- |
| `phaseChange` | ✅ | `—— 敌人出牌 ——` / `—— 你的回合 ——` / `—— 自动战斗 ——` / `—— 战斗结束 ——` |
| `energyReset` | ❌ | （静默） |
| `draw` | ❌ | （静默） |
| `drawSkipped` | ❌ | （静默） |
| `fatigue` | ✅ | `{谁}疲劳：受到 {damage} 点伤害，获得「疲劳突袭」（{generatedAttack} 攻）` |
| `playCard` | ✅ | `{谁}打出「{卡名}」`；若有目标则追加 `，目标：{目标称呼}` |
| `useSkill` | ✅ | `{谁}使用技能「{技能名}」`；若有目标则追加 `，目标：{目标称呼}` |
| `summon` | ✅ | `{谁}召唤了「{仆从名}」`（位置 index 一般不写，避免技术味） |
| `attack`（+ 可选下一事件 `counter`） | ✅ 合并一条 | 见下方 **交锋模板** |
| `counter` | ❌ 单独不成条 | 仅作为上一 `attack` 的配对载荷；孤立出现时见 §3.2.1 |
| `heal` | ✅ | `{目标}恢复 {amount} 点生命` |
| `death` | ✅ | `{谁}的「{仆从名}」阵亡` |
| `gameOver` | ✅ | `你获胜！` / `敌人获胜！` |

#### 3.2.1 交锋模板（`attack` + `counter`）

一次仆从交锋（自动战斗对撞或打脸）在日志中为**一条**：

```text
{发起方} X {目标}， {目标} -{attack.damage}HP，{发起方} -{counter.damage}HP
```

| 情况 | 文案 |
| --- | --- |
| 有反伤（下一事件为配对 `counter`） | `你的「铁壁守卫」 X 敌人， 敌人 -3HP，你的「铁壁守卫」 -2HP` |
| 无反伤（无 `counter`，如目标攻击力 0） | `你的「铁壁守卫」 X 敌人的「新兵」， 敌人的「新兵」 -1HP` |
| 仆从对仆从双向 | `敌人的「巨型魔像」 X 你的「铁壁守卫」， 你的「铁壁守卫」 -5HP，敌人的「巨型魔像」 -1HP` |

**配对判定（store 格式化时）：**

1. 遇到 `attack` 时窥视事件队列下一项；若为 `counter` 且 `counter.unit` 与 `attack.attacker` 为同一实体，则合并，且该 `counter` 播放时**不再追加日志**。
2. 否则按「无反伤」短句只根据 `attack` 写一条。
3. **孤立 `counter`**（当前直接攻击卡出牌路径：有 `playCard` + `counter`，无 `attack`）：本版不单独成条——出牌条已含目标；反伤数额若需可见，可后续把出牌条扩展为交锋句，或给该路径补发 `attack` 事件。第一版实现以「不刷孤立反伤条」为准。

**出牌与召唤：** 仆从卡会连续产生 `playCard` + `summon`。本版**两条都记**（先「打出」再「召唤」），语义清晰；若嫌重复，可后续合并为一条「打出「X」并召唤」。

**伤害法术：** `playCard`（含目标）后通常无 `attack`（法术单向扣血，无交锋事件）；伤害结果以出牌条 + 可能的 `death` 表达即可，不硬造交锋句。

### 3.3 命名解析时机

| 事件 | 解析依据 |
| --- | --- |
| `draw` / `playCard` | 用 `cardId` 在对应 side 的 deck/hand（或权威状态）找到 `defId` → `cardDb[defId].name` |
| `summon` / `death` | 用 `minionId` 在 **权威 board**（或事件应用前的 board）取 `minion` 显示名；死亡必须在从 view 移除**之前**解析名字 |
| `attack` / `counter` / `heal` | 用 `TargetRef`：hero → 称呼表；minion → board 上的 name；交锋条在消费 `attack` 时一次解析发起方与目标（及配对 `counter` 的承伤方，即发起方） |

解析失败时降级：`「未知卡牌」` / `「仆从」`，不抛错打断播放。

## 4. 怎么显示

### 4.1 布局位置与展开/收起

日志面板固定在视口**最左侧、垂直居中**（相对战场主区域左右布局：左栏日志、右栏战场），不放在战场下方。

```
┌────────┬────────────────────────────┐
│        │         敌方区域             │
│  日志  │         中线 / 能量          │
│  面板  │         己方区域 / 手牌       │
│        │                            │
└────────┴────────────────────────────┘
```

| 项 | 规格 |
| --- | --- |
| 位置 | 屏幕/战场布局**最左侧**，垂直方向大致**居中**对齐主战场 |
| 展开宽度 | 约 **200–260px**，高度约主战场中段（如 40–60vh 或对齐敌我仆从区高度），条目区内部纵向滚动 |
| 收起宽度 | 仅保留一条窄边/标签宽度（约 **32–40px**），不占决策视线 |
| 默认状态 | **展开** |
| 切换按钮 | 面板上提供**展开 / 收起**按钮（见下） |
| 层级 | 不挡住手牌、结束回合、选目标提示；窄屏可叠在战场左侧半透明，仍须可点收起 |

**展开 / 收起按钮：**

- 放在日志面板**靠战场一侧的边缘**（右侧），便于一眼找到。
- 展开时按钮文案/图标表示「收起」（如 `‹` / 「收起」）；收起时表示「展开」（如 `›` / 「日志」）。
- `aria-expanded` 反映当前状态；按钮 `aria-controls` 指向日志列表容器。
- 展开态切换可用短过渡（宽度/透明度），不必夸张动画。
- 展开状态**本版仅会话内 UI state**（组件 `useState` 即可），不要求写入 Zustand，也不持久化到 `localStorage`。

**列表行为（展开时）：**

- **自动滚到底**：每追加一条滚至最新（简单实现可总是滚到底）。
- **最新条高亮**（建议）：最新 1 条短暂提高对比度（约 1.2s），与事件播放同步感更强。
- 收起期间仍继续追加 log 数据；再次展开即可看到最新内容（展开时滚到底）。

### 4.2 视觉层级

按 `LogKind` 轻微区分颜色（沿用现有深色战场风，避免花哨）：

| kind | 建议 |
| --- | --- |
| `phase` | 略亮 / 居中或两侧加 `——`，作分隔 |
| `play` / `resource` | 默认次要文字色 |
| `combat` | 默认或略偏暖（受伤感） |
| `fatigue` | 警示色（如偏橙） |
| `system` | 强调色（胜负） |

敌我可用前缀着色：`你…` 偏蓝绿、`敌人…` 偏灰红——**仅文字色，不加卡片容器**（避免与战场「卡片」视觉抢戏）。

### 4.3 容量与生命周期

- 内存保留最近 **约 80～100 条**（当前实现约 40，建议略增）；超出丢弃最旧。
- `newGame` 清空日志并重置 `id` 序列。
- 不持久化到 `localStorage`（本版）。

### 4.4 无障碍

- 容器使用 `aria-label="战斗日志"`。
- 展开/收起按钮具备可理解的名称（如「收起战斗日志」/「展开战斗日志」）与 `aria-expanded`。
- 新条目可用 `aria-live="polite"` 播报**最新一条**（避免整表 live 刷屏）；若过吵，可仅对 `gameOver` / `fatigue` / `death` 播报。收起时 live 区域仍可更新，或仅在展开时播报（实现任选，优先少打扰）。

## 5. 完整回合示例

以下用现有示例卡（铁壁守卫、突击兵、火焰箭、治疗术等）串一条可读日志。左侧为引擎事件顺序，右侧为期望文案。

### 5.1 回合开始：敌人出牌 → 你的回合

| 事件 | 日志文案 |
| --- | --- |
| `phaseChange` → `enemyPlay` | `—— 敌人出牌 ——` |
| `draw` ×2（示例） | （静默，不写日志） |
| `playCard`（突击兵） | `敌人打出「突击兵」` |
| `summon` | `敌人召唤了「突击兵」` |
| `phaseChange` → `playerPlay` | `—— 你的回合 ——` |
| `draw` ×2 | （静默） |

### 5.2 玩家出牌：召唤 + 法术

| 事件 | 日志文案 |
| --- | --- |
| `playCard`（铁壁守卫） | `你打出「铁壁守卫」` |
| `summon` | `你召唤了「铁壁守卫」` |
| `playCard`（火焰箭，目标敌方突击兵） | `你打出「火焰箭」，目标：敌人的「突击兵」` |
| `death` | `敌人的「突击兵」阵亡` |

法术无 `attack`/`counter` 交锋对；结果看出牌目标 + 阵亡即可。

### 5.3 自动战斗：对撞 + 打脸（attack + counter → 一条）

| 事件（引擎仍可能两条） | 日志文案（一条） |
| --- | --- |
| `phaseChange` → `autoBattle` | `—— 自动战斗 ——` |
| `attack` + `counter`（守卫打脸，3 / 反伤 2） | `你的「铁壁守卫」 X 敌人， 敌人 -3HP，你的「铁壁守卫」 -2HP` |
| `attack` + `counter`（魔像打守卫，5 / 1） | `敌人的「巨型魔像」 X 你的「铁壁守卫」， 你的「铁壁守卫」 -5HP，敌人的「巨型魔像」 -1HP` |
| `death` | `你的「铁壁守卫」阵亡` |
| `attack` + `counter`（新兵打脸，1 / 2） | `敌人的「新兵」 X 你， 你 -1HP，敌人的「新兵」 -2HP` |
| `death` | `敌人的「新兵」阵亡` |

### 5.4 疲劳与跳过抽牌

| 事件 | 日志文案 |
| --- | --- |
| `drawSkipped` | （静默） |
| `fatigue`（damage=2, generatedAttack=1） | `你疲劳：受到 2 点伤害，获得「疲劳突袭」（1 攻）` |
| `fatigue`（再次，4 伤 / 2 攻） | `你疲劳：受到 4 点伤害，获得「疲劳突袭」（2 攻）` |

### 5.5 治疗与胜负

| 事件 | 日志文案 |
| --- | --- |
| `playCard`（治疗术，目标你） | `你打出「治疗术」，目标：你` |
| `heal` amount=2 | `你恢复 2 点生命` |
| `gameOver` winner=player | `你获胜！` |

### 5.6 一段连续日志（玩家视角读感）

```text
—— 敌人出牌 ——
敌人打出「突击兵」
敌人召唤了「突击兵」
—— 你的回合 ——
你打出「铁壁守卫」
你召唤了「铁壁守卫」
你打出「快速斩击」，目标：敌人的「突击兵」
敌人的「突击兵」阵亡
—— 自动战斗 ——
你的「铁壁守卫」 X 敌人， 敌人 -1HP，你的「铁壁守卫」 -2HP
—— 敌人出牌 ——
…
你获胜！
```

## 8. 验收标准（DoD）

- [x] 完整一局中，出牌、召唤、疲劳、交锋、治疗、死亡、阶段、胜负均有对应可读条目（抽牌不记）。
- [x] 自动战斗交锋：`attack`+`counter` 只出现**一条** `{谁} X {谁}， … -N HP` 文案。
- [x] 条目含卡名/仆从名（示例卡组下人工可读）。
- [x] 日志位于**最左侧中部**；有展开/收起按钮；**默认展开**；收起后不挡操作。
- [x] 日志与事件播放同步追加；`newGame` 清空。
- [x] format 函数有针对交锋合并与其它模板的单元测试。
